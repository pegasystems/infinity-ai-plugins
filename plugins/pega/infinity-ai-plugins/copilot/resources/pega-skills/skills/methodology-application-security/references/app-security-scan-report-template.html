Executive Summary

The current application manages regulated licensing workflows and includes sensitive personal and business data such as owner names, business addresses, contact numbers, email addresses, inspection notes, improvement notices, payment records, and uploaded documents. Based on the live rule inventory, the application’s security posture is below the expected standard for a production-facing regulated workflow.

The strongest verified control area is CORS, where both safe and unsafe policies exist and can be directly assessed. The weakest areas are application-specific authorization, authentication, encryption, and browser hardening: no app-specific ABAC controls, no discoverable app-specific authentication service rules, and no discoverable CSP artifacts were found in the visible rule set. Additionally, a service package for OAuth client management is configured with Basic authentication, which is not a strong default posture.

Verified Findings

Critical

No discoverable app-specific ABAC or property encryption policies

The scan found only 5 Rule-Access-Policy rules, all belonging to platform/system classes such as Work-Cover-Task and Data-OperatorPresence-Session. No application-specific access control policy was found for the licensing data model.

Why this matters: the application handles sensitive personal, business, and compliance data. Without app-specific ABAC and property encryption policies, there is no visible evidence of record-level restriction or field-level encryption for regulated data.

Recommended action: define Rule-Access-Policy and Rule-Access-PolicyCondition rules for licensing work and data classes, and add PropertyEncrypt policies for sensitive fields such as owner identity, contact details, payment-related fields, and inspection-sensitive fields.

Critical

Unsafe CORS policies are present in the environment

The scan verified multiple insecure CORS policy records:

  • WILDCARDCRED: wildcard origin * with pyAllowCredentials=true
  • INVALIDORIGINHTTP: insecure http://insecure-app.example.com with credentials enabled
  • LARGEMAXAGEABUSE: max age 2147483647
  • DUPLICATEORIGINS: duplicate trusted origins repeated three times
  • NOHEADERSCRED: credentials enabled with no explicit allowed headers present in the retrieved rule

Why this matters: these patterns can enable credential leakage, stale trust caching, operational confusion, and weak cross-origin governance.

Recommended action: remove wildcard-plus-credentials combinations, enforce HTTPS-only origins, cap max-age to a short interval such as 600 seconds, deduplicate origins, and define explicit allow/expose header lists when credentials are enabled.

Critical

OAuth client management service package still uses Basic authentication

The retrieved rule DATA-ADMIN-SERVICEPACKAGE OAUTHCLIENTS shows:

  • pyAuthenticationType: Basic
  • pyRequiresAuthentication: true
  • pyUseSSL: true
  • pyAccessGroup: PRPC:Administrators

Why this matters: Basic authentication is weaker than token-based or federated authentication for privileged administrative services. Even with TLS enabled, it increases exposure to credential replay, shared-secret misuse, and weaker credential governance.

Recommended action: move administrative and API-facing services to OAuth 2.0 or federated authentication, review access group scoping, and verify that secrets are not broadly distributed or statically embedded in integrations.

High

No discoverable app-specific authentication service rules

No Data-Admin-Security-AuthService rules were returned from the current accessible application context. That does not prove they do not exist elsewhere, but it does mean there is no visible evidence in the current rule inventory of app-specific SSO or stronger login configuration.

Recommended action: verify whether the application relies on an external enterprise IdP, and document the active authentication service, MFA policy, and requestor timeout settings.

High

No discoverable CSP artifacts for browser hardening

Searches for content security policy artifacts did not return CSP rule instances in the accessible scope. For a portal that may be accessed by vendors, officers, and administrators, absence of visible CSP configuration is a material browser-hardening concern.

Recommended action: create and assign Content Security Policy records with explicit directives such as default-src 'self', restrictive frame-ancestors, and narrow connect-src allowlists.

High

Document-heavy business flow with no verified malware scanning control

The application design explicitly includes document uploads for ID proof, address proof, photos, and layout plans. No discoverable evidence of attachment malware scanning or upload-hardening controls was obtained from the available artifacts.

Recommended action: verify antivirus integration for attachments, restrict file types, validate file signatures, and apply strict repository access controls.

High

Likely gap in record-level protection for regulator and vendor personas

The application model includes distinct personas with different access needs, including officers working within jurisdictions. No visible app-level ABAC policy enforces jurisdiction, ownership, or case-specific access filters.

Recommended action: implement ABAC conditions based on operator properties, ownership, jurisdiction, and exception lists for work and data classes.

Medium

Positive control: remediated CORS policy exists

The rule CORSNEW was verified in a safer state:

  • three explicit HTTPS origins
  • pyAllowCredentials=true with no wildcard origin
  • pyAllowedMethods=GET
  • pyMaxAge=600
  • TLS protocol version TLSv1.2

This is a good reference configuration for credentialed, narrowly scoped cross-origin access.

Medium

Positive control: secure CORS baseline pattern exists

The rule CLEANCORS shows an acceptable baseline pattern for non-credentialed cross-origin access with HTTPS origins, explicit allowed headers, explicit exposed headers, and pyAllowCredentials=false.

Medium

TLS is enabled on inspected service artifacts

The retrieved OAuth service package has pyUseSSL=true, and the inspected CORS rules carry pySSLProtocolVersion: TLSv1.2. This is positive, but it is not a substitute for stronger service authentication and application-level encryption.

Medium

Security governance evidence is not directly discoverable

The scan could not directly verify Security Checklist completion, Security Event Configuration state, dynamic security system settings, requestor timeout settings, or keystore activation through the exposed MCP tools.

Recommended action: manually review those landing pages in Dev Studio and record them in a control matrix.

Remediation Priorities

Immediate

  • Retire or isolate unsafe CORS policies such as WILDCARDCRED and INVALIDORIGINHTTP.
  • Replace Basic authentication on privileged service packages with OAuth 2.0 or federated auth.
  • Define ABAC policies for licensing work and data classes using ownership, jurisdiction, and role conditions.

Near Term

  • Create property encryption policies for sensitive fields and evaluate class-level encryption where appropriate.
  • Implement CSP records for all portals, especially if the application is internet-facing.
  • Verify Security Event Configuration for failed logins, security admin changes, OAuth events, and data access events.
  • Verify attachment virus scanning and upload hardening.

Validation Follow-up

  • Review Security Checklist completion in App Studio or Dev Studio.
  • Review keystore activation and external KMS posture on the Data Encryption landing page.
  • Review requestor timeout and MFA posture on security policies and access groups.

Conclusion

The application’s business model clearly requires stronger security than the currently discoverable configuration demonstrates. The environment contains both good and bad CORS examples, which is useful operationally, but it also exposes a more systemic problem: there is insufficient visible evidence of app-specific authorization, encryption, and browser-hardening controls for a regulated licensing workflow.

Overall assessment: Needs remediation before production-level trust.
